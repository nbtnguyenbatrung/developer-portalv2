
# 1. Introduction

VIETQRGlobal (cross border QR Code) is a cross-border money transfer service based on QRcodes. NAPAS QRcode Switching service provides technical supplement services to connect, transfer, transmit and handle electronic data for payment and related transactions among payment intermediaries, member banks and NAPAS’ partners using QR Code.

Customer can use mobile banking application to make payment transaction by scanning QR Code/ e-wallet to pay at merchants. The transaction will be proceeded by NAPAS and its switching network.

## 1.1.  Audience

The audience of this document are Payment Intermediaries, Member banks, Partners that connecting QRcode switching service.

## 1.2.  Scope

This document describes the general API contract, interfaces, data format, request and response messages of transactions to be used for integrating with NAPAS.

## 1.3. Support

For any assistance or information pertaining to existing or new QRcode switching service, kindly contact Napas Support.

## 1.5.  How to use this document

All technical APIs described in this document based on technical specification and function of QR Push payment

All API’s fields are described by their attributes:

* **Field Name:** Name of field as used to form the system call.
* **Field Type:** this field is numerical, alpha, etc.
* **Length:** length of field value
* **Request**: request message to Napas
* **Response**: response message from Napas
* **Description:** Field detail description.

Field indicator:

* O: Optional – data element is optional
* M: Mandatory: specify whether data element is required
* C: Conditional – presenting this field if some conditions are met.

# 4. Appendix A – Data Protection

The data exchange between Service consumer and NAPAS will be protected by 2 levels of encryption, including:

* _Transport level_: this is at the encrypting at transmission, using TLS protocol (TLS v1.2 or higher) or mTLS.
* _Business message encryption_: some sensitive data group in the business message will be encrypted and signed in JWE/JWS format for security between NAPAS and terminal or device where data is read. This level of encryption prevent the sensitive data can be seen by middle servers during transmissions.
* _Business message signature:_ refer to Message Signature

## 4.1. Certificate Scheme Exchange

Napas and partners should exchange the certificates during on-boarding process before making the service consuming.

Certificate information exchange:

* Certificate for handshake TLS – Certificate are used for trusting servers.
* Certificate for business message (For Napas, these certificates are the same):
  * Encryption data
  * Verify Signature

Summary about keys are used for data protection including business message encryption and signature:

* All asymmetric key sizes will be 2048-bit (RSA).
* All symmetric key sizes will be 128/256/512-bits (AES).
* Keys will be exchanged in X509 certificate format.

## 4.2. Message Signature

For client authentication and message integrity, every message transport to/from NAPAS APIs must contain a message signature individually. The signature is placed in header.signature of HTTP body request (for PUT/POST verb) or in header.signature in HTTP header property named header (for GET/DELETE).

### 4.2.1 Clients to NAPAS

* Client uses client’s private key which associated with Partner Certificate for business message signature to create message signature.
  * For POST/PUT methods:

header.signature = RSAwithSHA512(JSON\_minified(payload))

* * For GET/DELETE methods:

header.signature = RSAwithSHA512(header.requestor.id + header.reference-id + header.timestamp + header.operation)

* At server site, NAPAS will use client’s Partner Certificate for business message signature to verify the signature in the requested message.

### 4.2.2 NAPAS to Client

* NAPAS uses NAPAS’s private key which associated with NAPAS certificate for business message to create message signature.
  * For DELETE/GET/POST/PUT methods:

header.signature = RSAwithSHA512(JSON\_minified(result) + ”,” + JSON\_minified(payload))

* At client site, client shall use NAPAS certificate for business message to verify the signature in the response message.

# 5. Appendix B – JWE/JWS standard

Sensitive data is secured by using JWE/JWS format from device or terminal where data is read. This provide the end-to-end security solution which limit the risk of data is seen by any middle entities during transmission.

From Partner to encrypt data to Napas:

* Create JWE data using NAPAS certificate for business message CEK generate for every usage.
* Create JWS of JWE data using partner JWS private key (which will be verified by Partner Certificate for business message encryption)

From Partner to decrypt data from Napas:

* Verify JWS signature using NAPAS certificate for business message

Decrypt JWE payload data using partner JWE private key.
